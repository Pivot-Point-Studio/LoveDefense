# STAGE별 시나리오 유지 및 전환 규칙 구현 요구사항

## 0. 문서 목적

이 문서는 현재 게임에서 발생하는 다음 문제를 해결하기 위한 구현 요구사항이다.

- 같은 STAGE 안에서 AI가 임의로 새로운 상황을 만들어 대화의 분위기, 말투, 사건, 장소가 갑자기 바뀌는 문제
- 사용자에게 아무 안내 없이 다음 상황으로 넘어가 현재 무슨 일이 일어났는지 이해하기 어려운 문제
- 갈등이 일찍 해결됐다는 이유로 같은 STAGE 안에서 새 시나리오가 시작되는 문제
- STAGE 변경과 시나리오 변경의 시점이 분리되어 일관성이 깨지는 문제

이 문서의 규칙을 기준으로 기존 코드를 분석하고, 관련된 부분만 수정한다.

---

# 1. 최우선 핵심 규칙

## 1.1 한 STAGE에는 하나의 시나리오만 존재한다

> **1 STAGE = 1 Scenario**

STAGE가 시작될 때 생성되거나 불러온 하나의 시나리오는 해당 STAGE가 종료될 때까지 유지해야 한다.

같은 STAGE 안에서는 새로운 시나리오, 새로운 사건, 새로운 갈등 원인을 생성하면 안 된다.

### STAGE 동안 고정해야 하는 정보

- 시나리오 ID
- 상황 제목
- 시간대
- 장소
- 등장인물 관계
- 사용자 역할
- 상대방 역할
- 갈등의 근본 원인
- 현재 사건의 배경
- 상대방의 핵심 숨은 감정
- 상대방의 핵심 숨은 욕구
- 해당 STAGE의 대화 목표
- 해당 STAGE에서 사용하는 캐릭터 기본 설정
- 해당 STAGE에서 사용하는 의사소통 유형 및 애착 관련 설정

### STAGE 안에서 변경할 수 있는 정보

사용자 선택이나 자유 입력에 따라 아래 상태는 바뀔 수 있다.

- 상대방의 감정 강도
- 상대방의 말투
- 관계 HP
- 갈등 수치
- 신뢰도
- 화해 진행도
- 오해 해소 정도
- 상대방의 현재 반응
- 대화의 세부 흐름
- 사용자의 답변에 대한 평가 결과
- 현재 턴 번호

즉, **상황 자체는 고정하고 상황 안의 상태만 변화시킨다.**

# 2. STAGE 기본 규칙

## 2.1 기본 진행 구조

현재 기준으로 다음 구조를 사용한다.

- 기본 게임은 총 5 STAGE
- 기본적으로 1 STAGE는 5턴
- 1턴은 아래 흐름으로 구성한다.
  1. 상대방 발화 또는 상황 제시
  2. 사용자 선택 또는 자유 입력
  3. 사용자 입력 평가
  4. 관계 HP 및 갈등 수치 반영
  5. 상대방 반응 출력
  6. 현재 턴 증가

기존 코드에 STAGE 수 또는 턴 수를 관리하는 상수가 이미 있다면, 같은 값을 중복 하드코딩하지 말고 기존 단일 데이터 소스를 사용한다.

## 2.2 STAGE 난이도

STAGE가 높아질수록 난이도가 올라간다.

최소한 다음 요소를 반영한다.

- 클리어에 필요한 최소 관계 HP가 높아짐
- 허용되는 최대 갈등 수치가 낮아짐

실제 수치 계산 공식이 기존 코드에 존재한다면 해당 공식을 유지한다.

공식이 분산되어 있거나 중복되어 있다면 하나의 함수 또는 설정 객체에서 관리하도록 정리한다.

## 2.3 STAGE 시작 전 안내

각 STAGE 시작 전에는 다음 정보를 사용자에게 보여준다.

- 현재 STAGE 번호
- 해당 STAGE의 최소 요구 HP
- 해당 STAGE의 최대 허용 갈등 수치
- 이전 STAGE와 비교해 난이도가 어떻게 바뀌었는지

사용자가 확인 버튼을 누르기 전에는 해당 STAGE의 대화를 시작하지 않는다.

기존에 구현된 STAGE 중앙 팝업이 있다면 유지한다.

- 중앙에 `STAGE {번호}` 표시
- 약 3초간 표시
- 이후 STAGE 조건 안내 모달 표시 또는 기존 UI 흐름에 맞게 자연스럽게 연결

중앙 팝업과 조건 안내 모달의 표시 순서는 기존 UX와 충돌하지 않도록 하되, 사용자가 새 STAGE가 시작됐다는 사실을 명확히 알 수 있어야 한다.

---

# 3. 같은 STAGE 안에서 금지되는 동작

AI 또는 게임 로직은 STAGE가 끝나기 전 다음 행동을 하면 안 된다.

- 갑자기 새로운 장소로 이동
- 갑자기 시간대 변경
- 새로운 갈등 사건 추가
- 기존 갈등과 무관한 새로운 주제 시작
- 등장인물 관계 변경
- 캐릭터 성격 또는 의사소통 유형 변경
- 상대방이 이전 대화 내용을 잊은 것처럼 반응
- 화해가 되었다는 이유로 다른 시나리오 자동 생성
- 사용자가 요구하지 않았는데 장면 전환
- 시스템 승인 없이 다음 STAGE의 프롬프트 주입
- 현재 시나리오 객체를 새 객체로 덮어쓰기
- 현재 시나리오 ID를 같은 STAGE 중간에 변경

# 4. 갈등이 일찍 해결된 경우

## 4.1 기본 원칙

정해진 턴이 끝나기 전에 충분한 사과, 공감, 해명 또는 화해가 이루어져도 같은 STAGE 안에서는 새로운 시나리오로 넘어가지 않는다.

남은 턴은 같은 상황을 자연스럽게 마무리하는 데 사용한다.

## 4.2 남은 턴에서 사용할 수 있는 대화

- 상대방이 정말 안심했는지 확인
- 사용자가 상대방의 속뜻을 제대로 이해했는지 확인
- 감정 정리
- 오해가 완전히 풀렸는지 확인
- 앞으로 비슷한 상황에서 어떻게 행동할지 이야기
- 재발 방지 약속
- 감사 또는 안도 표현
- 관계 회복을 보여주는 자연스러운 후속 대화

## 4.3 해결 상태 관리 권장

필요하면 STAGE 내부 상태에 다음과 같은 값을 추가한다.

```js
resolutionState:
  | "unresolved"
  | "improving"
  | "resolved"
  | "worsened"
```

`resolutionState === "resolved"`여도 현재 STAGE의 남은 턴 동안 시나리오 ID와 배경은 변경하지 않는다.

AI에는 `resolved` 상태일 때 새로운 문제를 만들지 말고 후속 대화를 생성하도록 지시한다.

---

# 5. STAGE가 끝나는 조건

정상적인 STAGE 종료 조건은 다음과 같다.

- 해당 STAGE의 마지막 턴이 끝남
- 마지막 사용자 입력에 대한 평가가 완료됨
- 마지막 관계 HP 및 갈등 수치 반영이 완료됨
- 마지막 상대방 반응이 출력됨
- STAGE 성공 또는 실패 판정이 완료됨

STAGE 종료 판정은 사용자 입력을 받기 전에 실행하면 안 된다.

마지막 턴의 사용자 입력과 상대방 반응이 모두 처리된 뒤 실행한다.

---

# 6. STAGE 도중 종료가 허용되는 예외

다음 예외에서만 현재 STAGE를 즉시 종료할 수 있다.

## 6.1 게임 오버

- 관계 HP가 0 이하
- 갈등 수치가 게임 오버 한도를 초과
- 기존 게임 오버 조건을 충족

이 경우 새로운 시나리오를 생성하지 않고 게임 오버 화면 또는 결과 화면으로 이동한다.

## 6.2 사용자 직접 종료

- 홈으로 이동
- 뒤로가기 후 종료 확인
- 게임 종료 버튼 선택
- 세션 포기

이 경우 진행 중인 STAGE를 중단하고, 새 시나리오를 자동 생성하지 않는다.

## 6.3 개발자 테스트 기능

- 개발자 모드에서 강제로 다음 STAGE 이동
- 특정 STAGE 테스트
- 시나리오 재생성 테스트

개발자 기능은 일반 사용자 환경에서는 노출하지 않는다.

## 6.4 치명적인 데이터 오류

현재 시나리오 데이터가 없거나 필수 필드가 손상되어 진행이 불가능한 경우에만 안전한 복구 처리를 한다.

권장 처리 순서:

1. 동일 STAGE의 저장된 시나리오 복구 시도
2. 복구 실패 시 사용자에게 오류 안내
3. 현재 STAGE 재시작 선택 제공
4. 사용자 동의 없이 다음 STAGE로 넘기지 않음

단순히 AI 응답 하나가 실패했다는 이유로 새로운 시나리오를 생성하면 안 된다.

---

# 7. 다음 STAGE로 넘어가는 정확한 순서

현재 STAGE의 마지막 턴이 완료되면 다음 순서를 지킨다.

## 7.1 현재 STAGE 마무리

1. 사용자가 마지막 답변을 입력한다.
2. 사용자 답변을 평가한다.
3. 관계 HP와 갈등 수치를 반영한다.
4. AI가 현재 시나리오에 대한 마지막 상대방 반응을 생성한다.
5. 마지막 상대방 반응을 대화창에 표시한다.
6. STAGE 성공 또는 실패를 판정한다.
7. STAGE 결과를 저장한다.

## 7.2 현재 시나리오 종료 안내

대화창에 다음 시스템 메시지를 표시한다.

```text
📍 [System] 현재 시나리오가 종료되었습니다.
```

가능하면 일반 상대방 메시지와 시각적으로 구분한다.

권장 스타일:

- 중앙 정렬 또는 시스템 메시지 전용 정렬
- 작은 글씨
- 중립적인 배경색
- 상대방 프로필 이미지 없음
- 말풍선이 아닌 시스템 배지 형태 허용

## 7.3 STAGE 전환 안내

화면 중앙에 다음을 약 3초간 표시한다.

```text
STAGE {다음 번호}
```

기존 STAGE 팝업 구현이 있다면 재사용한다.

## 7.4 다음 STAGE 조건 안내

다음 STAGE 시작 전 모달 또는 전용 화면에 다음 내용을 표시한다.

```text
STAGE {다음 번호}

최소 요구 HP: {값}
최대 허용 갈등 수치: {값}

이전 STAGE보다
- 최소 요구 HP가 {증가값} 증가했습니다.
- 허용 갈등 수치가 {감소값} 감소했습니다.
```

사용자가 확인 버튼을 누르기 전에는:

- 새 시나리오를 대화창에 표시하지 않음
- 상대방의 첫 메시지를 생성하지 않음
- 입력창을 활성화하지 않음
- 턴 카운트를 시작하지 않음

## 7.5 새 시나리오 생성 및 표시

사용자가 확인 버튼을 누른 뒤에만 다음 작업을 한다.

1. 다음 STAGE 번호 확정
2. 새 시나리오 생성 또는 불러오기
3. 새 시나리오를 상태에 저장
4. 시나리오 ID 고정
5. 대화창에 시스템 메시지 표시
6. 새 상황 요약 표시
7. 상대방의 첫 발화 생성
8. 사용자 입력 활성화

대화창에 다음 메시지를 표시한다.

```text
📍 [System] 새로운 시나리오가 시작됩니다.
```

그 아래에 새 상황의 제목과 간단한 배경을 표시한다.

예시:

```text
상황: 연락 문제로 서운함이 쌓인 연인

최근 답장이 늦어지는 일이 반복되면서 상대방은 자신이 중요하지 않은 사람처럼 느끼고 있습니다.
```

이 안내가 표시된 뒤 상대방의 첫 메시지가 이어져야 한다.

# 8. 마지막 STAGE 종료

마지막 STAGE가 종료되면 다음 STAGE를 생성하지 않는다.

진행 순서:

1. 마지막 상대방 반응 출력
2. 최종 HP 및 갈등 수치 반영
3. 마지막 STAGE 결과 판정
4. 시스템 메시지 표시

```text
📍 [System] 현재 시나리오가 종료되었습니다.
```

5. 최종 게임 결과 계산
6. 결과 화면 또는 게임 오버 화면으로 이동
7. 랭킹 등록, 피드백 생성 등 기존 종료 로직 실행

마지막 STAGE 뒤에는 아래 메시지를 표시하면 안 된다.

```text
📍 [System] 새로운 시나리오가 시작됩니다.
```

# 9. AI 프롬프트에 반드시 추가할 규칙

현재 상대방 반응 생성 프롬프트 또는 시나리오 진행 프롬프트에 아래 의미가 반드시 포함되어야 한다.

```text
현재 STAGE가 종료되기 전에는 새로운 상황이나 사건을 생성하지 마십시오.

현재 시나리오의 시간, 장소, 갈등 원인, 등장인물 관계, 사용자 역할,
상대방 역할, 핵심 숨은 감정과 핵심 숨은 욕구는 STAGE가 끝날 때까지 유지해야 합니다.

사용자의 답변에 따라 상대방의 감정 강도, 말투, 신뢰도, 관계 HP,
갈등 수치, 화해 진행도와 대화의 세부 흐름만 변경할 수 있습니다.

현재 갈등이 일찍 해결되더라도 새로운 사건으로 넘어가지 마십시오.
같은 상황 안에서 화해 확인, 감정 정리, 오해 해소 확인,
재발 방지 약속 또는 관계 회복을 위한 후속 대화를 진행하십시오.

새로운 시나리오는 시스템이 새로운 STAGE 시작을 명시적으로 전달했을 때만 생성하십시오.

현재 STAGE의 scenarioId와 배경 정보를 변경하거나 재해석하지 마십시오.
기존 대화 기록과 모순되는 설정을 추가하지 마십시오.
```

## 9.1 AI에 전달해야 하는 현재 STAGE 정보

상대방 반응 생성 요청마다 최소한 다음 정보를 함께 전달한다.

```json
{
  "stageNumber": 1,
  "turnNumber": 3,
  "maxTurns": 5,
  "scenarioId": "scenario_xxx",
  "scenarioTitle": "면접 결과를 기다리는 연인",
  "contextSummary": "면접 결과 연락을 기다리며 불안해하고 있다.",
  "location": "각자의 집에서 메신저로 대화 중",
  "timeContext": "늦은 밤",
  "conflictCause": "결과를 기다리는 불안과 정서적 지지 부족",
  "hiddenEmotion": "불안하고 초조하다.",
  "hiddenNeed": "해결책보다 먼저 감정을 인정받고 싶다.",
  "resolutionState": "improving",
  "relationshipHp": 72,
  "conflictLevel": 24,
  "conversationHistory": []
}
```

필드 이름은 기존 코드 스타일에 맞게 조정할 수 있다.

중요한 것은 매 턴마다 동일한 `scenarioId`와 고정 배경이 전달되는 것이다.

# 10. 상태 관리 권장 구조

기존 상태 구조를 우선 사용하되, 아래와 같은 개념이 분리되어 있어야 한다.

```js
gameState = {
  currentStage: 1,
  totalStages: 5,
  currentTurn: 1,
  turnsPerStage: 5,

  currentScenario: {
    id: "scenario_xxx",
    title: "...",
    contextSummary: "...",
    location: "...",
    timeContext: "...",
    conflictCause: "...",
    hiddenEmotion: "...",
    hiddenNeed: "...",
    userRole: "...",
    partnerRole: "...",
    goal: "..."
  },

  stageStatus: "playing",
  resolutionState: "unresolved",

  relationshipHp: 0,
  conflictLevel: 0
};
```

`stageStatus` 권장 값:

```js
"stage_intro"
"playing"
"stage_resolving"
"stage_result"
"stage_transition"
"game_result"
"game_over"
```

## 10.1 상태 전환 예시

```text
stage_intro
  -> 사용자가 확인
playing
  -> 마지막 턴 완료
stage_resolving
  -> 최종 반응 및 점수 반영 완료
stage_result
  -> 결과 저장 완료
stage_transition
  -> 다음 STAGE 안내 및 새 시나리오 준비
stage_intro
```

마지막 STAGE인 경우:

```text
stage_result
  -> game_result 또는 game_over
```

## 10.2 중복 호출 방지

다음 STAGE 생성 함수가 두 번 호출되지 않도록 잠금 또는 상태 검사를 추가한다.

예시:

```js
if (gameState.stageStatus !== "stage_transition") {
  return;
}

if (gameState.isGeneratingScenario) {
  return;
}
```

사용자가 확인 버튼을 여러 번 눌러도 시나리오 생성 API가 중복 호출되면 안 된다.

# 11. 시스템 메시지 UI 규칙

시스템 메시지는 일반 채팅 메시지와 분명히 구분한다.

## 필수 메시지

### 현재 시나리오 종료

```text
📍 [System] 현재 시나리오가 종료되었습니다.
```

### 새 시나리오 시작

```text
📍 [System] 새로운 시나리오가 시작됩니다.
```

## 권장 접근성 및 표시 규칙

- 색상만으로 의미를 구분하지 않는다.
- `[System]` 텍스트를 유지한다.
- 스크린리더가 읽을 수 있는 실제 텍스트로 렌더링한다.
- 시스템 메시지 표시 후 채팅창이 해당 메시지로 자동 스크롤된다.
- 새 시나리오 요약과 상대방 첫 발화가 너무 빠르게 동시에 나타나 사용자가 놓치지 않도록 한다.
- 기존 애니메이션이 있다면 과도하게 길게 만들지 않는다.

# 12. 저장 및 복원 규칙

게임을 새로고침하거나 잠시 이탈한 뒤 복원할 때도 같은 STAGE의 시나리오가 유지되어야 한다.

가능하면 다음 값을 저장한다.

- currentStage
- currentTurn
- currentScenario 전체 또는 scenarioId와 복원 가능한 데이터
- relationshipHp
- conflictLevel
- resolutionState
- stageStatus
- 대화 기록
- 이미 처리된 마지막 사용자 입력 ID
- 이미 처리된 STAGE 종료 여부

새로고침 후 같은 STAGE를 복원할 때 새 시나리오 생성 API를 호출하면 안 된다.

현재 STAGE에 저장된 시나리오가 있으면 그대로 사용한다.

# 13. 오류 처리 규칙

## 13.1 AI 응답 실패

AI 반응 생성이 실패했을 때:

- 현재 시나리오를 유지한다.
- 새 시나리오를 생성하지 않는다.
- 같은 요청 재시도 기능을 제공한다.
- 중복 점수 반영을 막는다.
- 실패한 턴을 완료 처리하지 않는다.

## 13.2 새 시나리오 생성 실패

다음 STAGE의 새 시나리오 생성이 실패했을 때:

- 현재 STAGE 결과는 유지한다.
- 다음 STAGE 번호를 임의로 더 증가시키지 않는다.
- 사용자에게 재시도 UI를 제공한다.
- 재시도 시 같은 다음 STAGE 번호로 생성한다.
- 생성 성공 전까지 사용자 입력창을 활성화하지 않는다.

# 15. Codex 작업 지시

## 15.1 먼저 수행할 분석

코드를 수정하기 전에 다음을 찾고 간단히 정리한다.

- STAGE 번호를 관리하는 파일과 변수
- 턴 수를 관리하는 파일과 변수
- 현재 시나리오를 생성하는 함수
- 상대방 반응을 생성하는 함수
- OpenAI API 또는 Supabase Edge Function 호출 위치
- STAGE 종료 판정 위치
- 다음 STAGE 이동 함수
- 시스템 메시지 렌더링 함수
- STAGE 중앙 팝업 코드
- 관계 HP 및 갈등 수치 반영 위치
- 저장 및 복원 로직
- 관련 테스트 코드

## 15.2 구현 원칙

- 기존 구조를 최대한 유지한다.
- 동일 기능을 새로 중복 구현하지 않는다.
- STAGE 전환 로직은 가능한 한 하나의 함수 또는 하나의 명확한 흐름으로 통합한다.
- 시나리오 생성과 상대방 반응 생성을 분리한다.
- 상대방 반응 생성 함수가 새 시나리오를 만들지 못하게 한다.
- 새 시나리오 생성은 다음 STAGE 시작 시점에만 호출한다.
- 비동기 중복 호출과 버튼 중복 클릭을 방지한다.
- 기존 변수명과 코드 스타일을 따른다.
- 관련 없는 파일은 수정하지 않는다.

## 15.3 구현 후 보고할 내용

작업 완료 후 다음을 알려준다.

1. 수정한 파일 목록
2. 각 파일에서 변경한 핵심 내용
3. STAGE와 시나리오가 어떤 상태 흐름으로 동작하는지
4. AI 프롬프트에 추가한 제한
5. 중복 호출을 막은 방법
6. 새로 추가하거나 수정한 테스트
7. 사용자가 직접 확인해야 하는 수동 테스트 절차
8. 남아 있는 위험 요소 또는 미완성 부분

# 16. 필수 테스트 시나리오

## 테스트 1. 같은 STAGE에서 상황 유지

1. Stage 1 시작
2. 첫 시나리오의 장소, 시간, 갈등 원인 기록

4. 모든 AI 반응 확인

기대 결과:

- 새로운 사건이 시작되지 않음

## 테스트 2. 3턴 만에 화해

1. Stage 1 시작
2. 사용자가 3턴 안에 매우 적절한 답변 선택
3. 해결 상태를 `resolved`로 만듦

기대 결과:

- Stage 1 안에서 새 시나리오가 생성되지 않음
- 남은 턴은 감정 정리와 관계 회복 대화
- Stage 1의 마지막 턴 뒤에만 전환

## 테스트 3. 마지막 턴 전환 순서

1. Stage 1의 마지막 사용자 입력

기대 순서:

마지막 상대방 반응

# 17. 완료 조건

다음 조건을 모두 만족해야 작업 완료로 본다.

- 한 STAGE 안에서 scenarioId가 바뀌지 않는다.
- AI가 같은 STAGE 중간에 새 사건을 생성하지 않는다.
- 갈등이 일찍 해결돼도 남은 턴은 같은 상황의 마무리로 진행된다.
- 마지막 턴의 상대방 반응이 출력된 뒤에만 STAGE 종료 처리가 시작된다.
- 현재 시나리오 종료 시스템 메시지가 표시된다.
- 다음 STAGE 중앙 팝업이 표시된다.
- 다음 STAGE 조건 안내가 표시된다.
- 사용자 확인 전에는 새 시나리오가 시작되지 않는다.
- 사용자 확인 후 새 시나리오 시작 시스템 메시지가 표시된다.
- 새 상황 요약 뒤에 상대방 첫 발화가 표시된다.
- 마지막 STAGE 뒤에는 새 시나리오를 생성하지 않는다.
- API 실패나 새로고침 후에도 현재 시나리오가 유지된다.
- 중복 클릭으로 STAGE나 시나리오가 중복 생성되지 않는다.
- 기존 HP, 갈등, 평가, 결과, 랭킹 기능이 정상 동작한다.
- 관련 자동 테스트 또는 수동 테스트 절차가 준비되어 있다.

# 18. 최종 구현 요약

최종적으로 게임은 아래 구조로 동작해야 한다.

```text
STAGE 시작
-> 새 시나리오 1개 생성 및 고정
-> 같은 상황으로 정해진 턴 진행
-> 감정, 말투, HP, 갈등 수치만 변화
-> 마지막 턴의 상대방 반응 출력
-> STAGE 결과 판정
-> "현재 시나리오가 종료되었습니다."
-> 다음 STAGE 중앙 팝업
-> 다음 STAGE 조건 안내
-> 사용자 확인
-> "새로운 시나리오가 시작됩니다."
-> 새 상황 요약
-> 다음 STAGE의 첫 상대방 발화
```

핵심은 아래 한 문장이다.

> **새로운 시나리오는 STAGE가 바뀌고 사용자가 다음 STAGE 시작을 확인한 뒤에만 생성한다.**
