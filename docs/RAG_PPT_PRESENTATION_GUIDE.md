# 턴별 AI 피드백 RAG 발표 가이드

## 1. 발표에서 사용할 정확한 한 문장

연애 디펜스는 사용자·상대 성향, 현재 갈등, 최근 대화, 이번 사용자 답변을 검색 질의로 구성하고, 서버의 프로젝트 큐레이션 코칭 지식 카드 중 관련도 높은 Top-3를 검색하여 AI 피드백과 추천 표현 생성에 추가한다.

현재 구현은 **프로젝트 큐레이션 지식 카드 기반 metadata·keyword retrieval RAG**다. 임베딩이나 Vector DB를 사용하는 Semantic Vector RAG는 아직 아니다.

---

## 2. 왜 RAG를 적용했는가

기존 방식은 현재 Stage와 대화 내용을 OpenAI에 직접 전달했다. 이 방식은 대화 문맥을 반영할 수 있지만 다음 문제가 있다.

- 같은 유형의 갈등에서도 추천 전략이 일관되지 않을 수 있다.
- 추천 표현이 일반적인 공감 문장으로 수렴할 수 있다.
- 어떤 코칭 원칙이 사용되었는지 추적하기 어렵다.
- 프로필을 프롬프트에 넣는 것만으로는 RAG라고 할 수 없다.

RAG 적용 후에는 답변 생성 전에 관련 코칭 원칙을 검색하고, 검색 결과와 식별자를 함께 남긴다. 따라서 추천 표현이 어떤 지식 카드에 근거했는지 확인할 수 있다.

---

## 3. 현재 구현 범위

### 적용 대상

- 매 턴 사용자 답변 평가
- AI 피드백의 아쉬운 부분
- `suggestedBetterResponse` 추천 표현
- 피드백 메시지의 `RAG 참고 근거` 표시

### 적용하지 않은 대상

- Stage 클리어 공식
- 관계 HP·갈등 변화 공식
- 상대방의 고정된 사실이나 Stage 배경
- 로컬 fallback 평가
- Vector DB와 임베딩 기반 의미 검색
- 외부 연구 자료 자동 수집

상대방 대사와 평가를 한 번에 생성하는 통합 요청에서도 검색 Context가 전달되지만, 프롬프트는 검색 자료를 추천 표현과 평가 근거에만 사용하고 상대방 대사가 코치 말투로 변하지 않도록 분리한다.

---

## 4. 구현 구조

```text
사용자 답변
  + 사용자 성향 스냅샷
  + 상대 성향 스냅샷
  + 현재 Stage·갈등 원인
  + 숨은 감정·욕구
  + 최근 대화
        │
        ▼
서버 검색 Query 구성
        │
        ▼
큐레이션 코칭 카드 관련도 계산
        │
        ▼
Top-3 검색
        │
        ├── 검색 지식 → OpenAI 평가 Prompt
        └── 검색 ID·관련도 → 응답 Metadata
                              │
                              ▼
                피드백·추천 표현·RAG 근거 표시
```

### 주요 파일

| 역할 | 파일 |
| --- | --- |
| 큐레이션 코칭 지식 카드 | `supabase/functions/evaluate-turn/coachingKnowledge.js` |
| 검색 Query·관련도 계산·Top-K 검색 | `supabase/functions/evaluate-turn/ragRetriever.js` |
| 검색 후 OpenAI Prompt 증강 | `supabase/functions/evaluate-turn/index.ts` |
| 사용자·상대 프로필 전달 | `src/App.jsx`, `src/game/stagePromptContext.js` |
| 검색 Metadata 전달 | `src/backend/integrations/aiTurnService.js`, `src/game/turnEvaluationService.js` |
| 피드백 근거 UI | `src/App.jsx`, `src/index.css` |
| 검색 회귀 테스트 | `src/game/ragRetriever.test.js` |

## 5. 검색 데이터와 검색 방식

현재 지식베이스에는 다음 유형의 프로젝트 큐레이션 Seed 카드가 포함된다.

- 해결책보다 감정 확인 먼저 하기
- 숨은 욕구 확인하기
- 비난 대신 관찰과 영향 말하기
- 작고 구체적인 행동 요청
- 영향을 인정하는 사과와 관계 회복
- 안전한 대화 중단과 재개 시점 합의
- 친구·제3자 관계 경계 합의
- 금전 갈등의 숫자·일정 합의
- 답장 지연에서 사실과 불안 분리
- 귀가 등 안전 문제 우선 처리
- 감정 강도가 높을 때 갈등 완화
- 요약 확인 후 자신의 입장 말하기

검색기는 현재 문맥에서 주제, 키워드, 감정, 욕구, 프로필 성향, 최근 대화 신호를 추출해 관련도를 계산한다. 가장 관련 있는 카드 최대 3개를 OpenAI 입력에 추가한다.

검색 전략 식별자는 다음과 같다.

```text
curated_metadata_keyword_rag_v1
```

이 명칭을 발표 자료나 로그 화면에 그대로 사용하면 Vector RAG와 혼동되는 것을 막을 수 있다.

---

## 6. 개인정보와 추적성

- 검색은 Supabase Edge Function 내부에서 수행한다.
- 검색 Metadata에는 사용자 답변 원문을 저장하지 않는다.
- 원문 대신 `queryFingerprint`를 남긴다.
- 피드백에는 검색된 지식 카드 ID, 제목, 관련도, 지식베이스 버전이 연결된다.
- 지식 카드의 출처 유형은 `project_curated_seed`로 표시한다.

현재 카드는 프로젝트 내부 코칭 원칙이므로 외부 전문가 검수나 연구 근거를 주장해서는 안 된다.

## 7. PPT 권장 구성

### 슬라이드 1 — 문제 정의

제목 예시:

> 문맥만 전달하는 AI에서, 근거를 검색하는 AI로

포함할 내용:

- 기존에는 프로필·대화·Stage를 바로 OpenAI에 전달
- 추천 표현이 일반적이거나 일관되지 않을 가능성
- 생성 전에 관련 코칭 전략을 선택할 필요

### 슬라이드 2 — RAG 적용 전·후

왼쪽에는 기존 흐름을 표시한다.

```text
프로필 + 대화 → OpenAI → 피드백
```

오른쪽에는 적용 후 흐름을 표시한다.

```text
프로필 + 대화
      ↓
코칭 지식 Top-3 검색
      ↓
검색 Context + OpenAI
      ↓
근거가 연결된 피드백
```

검색 단계만 강조색으로 표시하면 RAG의 차이가 명확해진다.

### 슬라이드 3 — 실제 검색 과정

예시 입력:

```text
상대 성향: 답장이 늦으면 불안해지는 편
갈등: 연락과 답장 지연
사용자 답변: 바빴다니까 왜 자꾸 연락해?
```

검색 결과 표 예시:

| 순위 | 검색 지식 | 관련 신호 | 관련도 |
| --- | --- | --- | --- |
| 1 | 답장 지연에서 사실과 불안 분리 | 답장, 연락, 불안 | 실제 실행값 사용 |
| 2 | 해결책보다 감정 확인 먼저 하기 | 감정, 공감 | 실제 실행값 사용 |
| 3 | 비난 대신 관찰과 영향 말하기 | 비난, 책임 | 실제 실행값 사용 |

발표 자료에는 코드에 적힌 예측값을 쓰지 말고 실제 실행 응답의 `relevanceScore`를 캡처하여 사용한다.

### 슬라이드 4 — Prompt 증강

전체 Prompt를 작게 붙이지 말고 다음 세 부분만 확대한다.

1. 현재 대화 문맥
2. `retrievedCoachingContext`
3. 검색 지식을 추천 표현에만 사용하라는 제약

강조 문구:

> 검색된 자료는 상대방의 사실이 아니라 피드백 생성용 코칭 근거로만 사용한다.

### 슬라이드 5 — 동일 입력 RAG OFF/ON 비교

동일한 사용자 답변을 고정하고 결과를 나란히 배치한다.

| RAG OFF | RAG ON |
| --- | --- |
| 일반적인 공감 권유 | 현재 갈등에 맞는 감정 확인·행동 요청 |
| 사용 근거 확인 불가 | 지식 카드 ID와 관련도 확인 가능 |
| 결과 추적 어려움 | 지식베이스 버전과 Query 지문 기록 |

실제 A/B 실행 결과를 사용해야 하며, 기대 문구를 실제 결과처럼 제시하지 않는다.

### 슬라이드 6 — 사용자 화면

AI 피드백 메시지 아래의 `RAG 참고 근거` 칩을 캡처한다.

강조할 내용:

- 어떤 코칭 카드가 검색됐는지 사용자도 확인 가능
- 추천 표현 저장 기능과 연결
- 관련도는 의미 유사도나 정답 확률이 아니라 현재 검색 전략의 상대적 관련도

### 슬라이드 7 — 검증과 관찰 가능성

현재 증거:

- 답장 지연 문맥 검색 테스트
- 금전 갈등 검색 우선순위 테스트
- 검색 Metadata에 원문이 포함되지 않는 테스트
- 전체 게임 단위 테스트 통과 결과

추가 측정 권장 지표:

- 사람이 판단한 Top-3 검색 적합률
- 추천 표현과 검색 지식의 근거 일치율
- 일반적·반복적 추천 표현 비율
- RAG OFF/ON 추천 표현 선호도
- 검색 및 전체 응답 지연 시간
- 검색 카드별 사용 빈도 편향

측정하지 않은 수치를 임의로 만들어 발표하지 않는다.

### 슬라이드 8 — 한계와 다음 단계

현재 한계:

- 내부 큐레이션 Seed 카드 기반이며 사람 검수·외부 연구 근거가 아직 없음
- metadata·keyword 검색이므로 표현이 크게 달라지면 검색 품질이 낮아질 수 있음
- OpenAI 요청 실패 후 로컬 fallback에는 RAG 추천이 적용되지 않음
- 검색 카드 수와 주제가 제한적임

다음 단계:

1. 출처와 사용 권한이 확인된 관계 커뮤니케이션 자료 추가
2. 문서 Chunk와 출처 Metadata 관리
3. 임베딩 생성과 Supabase `pgvector` 도입
4. Keyword + Vector Hybrid Search 비교
5. 검색 평가용 골든 Query 세트 구축

## 8. 데모 순서

1. 프로필에서 사용자와 상대방 성향을 다르게 설정한다.
2. 답장 지연 또는 금전 갈등 Stage를 실행한다.
3. 문맥과 충돌하는 사용자 답변을 입력한다.
4. 채팅 메뉴에서 `AI 피드백 보기`를 선택한다.
5. 추천 표현과 `RAG 참고 근거`를 함께 보여준다.
6. 네트워크 응답 또는 저장된 턴 결과에서 다음 값을 보여준다.

```text
retrieval.strategy
retrieval.knowledgeBaseVersion
retrieval.queryFingerprint
retrieval.sources[].knowledgeId
retrieval.sources[].relevanceScore
```

7. 같은 입력의 RAG OFF 결과가 준비돼 있다면 나란히 비교한다.

Edge Function을 배포하지 않은 로컬 프론트엔드만으로는 서버 RAG가 실행되지 않는다. 발표 전 `evaluate-turn` 함수가 최신 코드로 배포되어 있는지 확인한다.

## 9. 추천 발표 문구

### 기술 설명

> 매 턴 사용자 답변을 바로 생성 모델에 보내지 않고, 사용자·상대 성향과 현재 갈등 문맥으로 검색 질의를 만든 뒤 프로젝트 큐레이션 코칭 지식 Top-3를 검색합니다.

> 검색된 지식은 AI의 점수를 강제로 결정하는 정답지가 아니라, 아쉬운 부분과 추천 표현을 구체화하는 제한된 근거로 사용합니다.

> 검색 결과의 카드 ID, 관련도, 지식베이스 버전을 응답에 함께 남겨 어떤 근거가 사용됐는지 추적할 수 있습니다.

> 현재 버전은 metadata·keyword 기반 RAG이며, 다음 단계에서 사람 검수와 출처 확인을 마친 외부 자료와 Vector Search를 추가할 수 있도록 검색 계층을 분리했습니다.

### 사용자 가치

> 같은 공감 문장을 반복하는 대신, 답장 지연·금전·관계 경계처럼 현재 갈등에 맞는 대화 전략을 찾아 추천합니다.

> 사용자는 추천 표현뿐 아니라 어떤 코칭 원칙이 참고됐는지도 확인할 수 있습니다.

### 안전성과 정직성

> 검색 카드에는 사용자 원문을 저장하지 않고 Query 지문만 남기며, 현재 지식은 프로젝트 내부 큐레이션 Seed라는 출처 유형을 명확히 표시합니다.

## 10. 피해야 할 발표 문구

### 구현하지 않은 기술을 주장하는 표현

> OpenAI Embedding과 Vector DB로 의미 검색을 구현했습니다.

현재 구현에는 임베딩과 Vector DB가 없으므로 사용하면 안 된다.

> Supabase pgvector에서 유사 문서를 검색합니다.

현재는 Edge Function 내부 지식 카드 검색이므로 사실이 아니다.

### RAG가 아닌 기능을 RAG로 설명하는 표현

> 사용자 프로필과 최근 대화를 Prompt에 넣었기 때문에 RAG입니다.

프로필과 최근 대화를 직접 전달하는 것은 문맥 제공이며 검색 증강이 아니다.

> 내부 사례를 무작위로 선택하므로 RAG입니다.

무작위 선택은 질의 기반 검색이 아니다.

### 근거 수준을 과장하는 표현

> 전문 상담사가 검증한 심리학 기반 조언입니다.

현재 지식 카드는 프로젝트 내부 Seed 데이터이며 전문가 검수 근거가 없다.

> 논문으로 증명된 최적의 연애 답변을 제공합니다.

검색 결과와 추천 표현은 가능한 코칭 전략 중 하나이며 정답이 아니다.

> 관련도 90%이므로 답변 정확도가 90%입니다.

`relevanceScore`는 현재 검색 알고리즘 내부의 상대적 관련도이지 정확도·확률·임베딩 유사도가 아니다.

### 제품 범위를 과장하는 표현

> 모든 상대 대사와 게임 시나리오가 RAG로 생성됩니다.

현재 RAG의 핵심 적용 대상은 턴별 피드백과 추천 표현이다.

> RAG가 실제 사용자의 성격을 정확히 진단합니다.

프로필은 개인화를 위한 참고 정보일 뿐 진단 결과가 아니다.
